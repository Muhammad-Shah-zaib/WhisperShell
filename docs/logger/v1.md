# WhisperShell V1 Development Log
**Date:** 2026-07-10

## 1. Project Initialization & Validation Phase
Before building the actual Tauri application, we created a standalone CLI validation tool (`tests/whisper_validation`). 
- **Goal:** Validate the core pipeline: Microphone Capture (`cpal`) -> Resampling (`rubato`, 48kHz to 16kHz) -> AI Transcription (`whisper-rs`).
- **GPU Acceleration:** We bypassed the NVIDIA CUDA compiler (which was failing due to GCC 14 constraints on Fedora 44) and successfully implemented the **Vulkan backend**. We also implemented hybrid GPU detection to ensure the model targets the NVIDIA discrete GPU (Device 1) rather than the Intel iGPU (Device 0).
- **Result:** Successfully transcribed audio locally in real-time. Documented in `tests/SESSION_SUMMARY.md`.

## 2. V1 Tauri Application Development
We wrapped the tested pipeline in a Tauri v2 application inside the `src-tauri` directory.
- **Architecture & Design Patterns:**
  - **Single-Process Daemon:** The app runs as a resident background process with a minimal HTML/CSS/JS frontend (served directly from `ui/`).
  - **Thread-safe State Management:** Implemented an `AppState` struct encapsulated in an `Arc<Mutex<AppState>>` in Rust to handle concurrent recording, processing, and UI events safely.
  - **Event-Driven UI:** Used Tauri's `Emitter` to push state changes (`Recording`, `Processing`, `Idle`, and transcripts) directly to the frontend for reactive UI updates without polling.
  - **Atomic Clipboard Injection:** Rather than simulating keystrokes character-by-character (which is often flaky on Wayland due to async input events), we bypassed keystrokes completely and directly wrote the transcribed text to the system clipboard using the `arboard` crate.

## 3. Wayland Global Hotkey Resolution
- **Issue:** The `tauri-plugin-global-shortcut` successfully registered our hotkey (`Ctrl + Space`) but failed to capture keypress events natively on the GNOME Wayland desktop, causing the UI to stay stuck in Idle.
- **Root Cause:** Native Wayland compositors (like Mutter on GNOME) explicitly sandbox applications and block global key grabs for security reasons. The underlying plugin relies on X11's `XGrabKey` API.
- **Technique / Solution:** We forced the application to run under XWayland by injecting `std::env::set_var("GDK_BACKEND", "x11");` inside `main.rs` *before* GTK initialization. This routes the app's windowing context through the X11 compatibility layer (XWayland), which restores full support for global hotkey grabbing on Linux.
- **Observability:** We also added detailed terminal logging (verbose prints) inside the global hotkey handler to trace raw shortcut events and track state transitions seamlessly.

## 4. Overlay HUD Window (Visual Feedback)

### 4.1 Goal
Build a transparent, always-on-top overlay HUD anchored at the bottom-center of the screen that shows three states: **Listening** (voicegrams), **Processing** (spinner), and **Done** (transcript copied, auto-dismiss after 2s).

### 4.2 Architecture Decision — Second Tauri Window (Not Layer Shell)
- **Considered:** `wlr-layer-shell` for a proper compositor-pinned overlay (like a notification daemon would use).
- **Rejected:** GNOME/Mutter does not implement `wlr-layer-shell`. This is the same dead end Handy hit.
- **Chosen Solution:** A second Tauri `WebviewWindow` pre-declared in `tauri.conf.json` with `decorations: false`, `transparent: true`, `alwaysOnTop: true`, `visibleOnAllWorkspaces: true`, `skipTaskbar: true`. This works because the app already runs under **XWayland** (`GDK_BACKEND=x11`), and GTK's X11 backend fully supports all those properties.
- **Focus safety:** `focus: false` in config + explicit `set_always_on_top(true)` on every `show()` call ensures the overlay never steals keyboard focus from whatever the user is typing into.

### 4.3 Files Created / Modified
- **`src-tauri/ui/overlay.html`** — Self-contained pill-shaped glassmorphism card. Listens for Tauri events (`state_changed`, `transcript_ready`) and switches between three CSS-animated states. Voicegrams are 9 thin CSS bars (`2.5px` wide) with staggered `alternate` keyframe animation; no mic input sync required by design.
- **`tauri.conf.json`** — Added overlay window entry (`320×80`, hidden at startup, `visibleOnAllWorkspaces: true`).
- **`capabilities/default.json`** — Added `"overlay"` to windows list; added `core:window:allow-hide/show/center`.
- **`src-tauri/src/lib.rs`** — Added `show_overlay()` / `hide_overlay()` helpers using `app.get_webview_window("overlay")`. Position computed with pure physical-pixel math using `monitor.position()` + `monitor.size()` + `monitor.scale_factor()` — multi-monitor safe. Auto-hide spawns a background thread with `thread::sleep(2s)`.

### 4.4 Positioning Fix
- **Initial bug:** Overlay appeared slightly left of center. Root cause: calling `overlay.center()` (async under GTK) then immediately reading `outer_position()` returned stale coordinates before the center operation completed.
- **Fix:** Removed `center()` entirely. Now uses fully synchronous arithmetic: `x = monitor_x + (monitor_w - win_w) / 2`, `y = monitor_y + monitor_h - win_h - margin`. Accounts for HiDPI via `scale_factor`.

## 5. Global Hotkey — XGrabKey Failure & evdev Migration

### 5.1 Root Cause Analysis
After adding the overlay and testing cross-workspace behaviour, the hotkey stopped firing whenever a **native Wayland window** had focus (terminal, browser, Files, etc.).

- **Diagnosis:** The `tauri-plugin-global-shortcut` plugin uses X11's `XGrabKey` API under the hood. On GNOME + Wayland, even with `GDK_BACKEND=x11` (XWayland), Mutter only forwards X11 passive key grabs to XWayland clients **when an X11 client has focus**. When a native Wayland compositor client has focus, the keystroke is consumed entirely by the Wayland input dispatch path and never reaches XWayland — so `XGrabKey` silently does nothing.
- **This is fundamental to Wayland's security model** — compositors explicitly prevent apps from eavesdropping on other apps' keystrokes. `XGrabKey` is an X11 escape hatch that only partially works under XWayland.
- **Confirmed:** GNOME does NOT implement the `xdg-desktop-portal` GlobalShortcuts interface (only KDE/Hyprland do), so there is no approved Wayland-native global shortcut API available.

### 5.2 Solution — evdev Kernel-Level Input
- **Approach:** Read raw keyboard events directly from `/dev/input/eventN` via the Linux `evdev` subsystem. This reads hardware events at kernel level, completely bypassing both Wayland and X11. Focus and workspace are irrelevant.
- **This is identical to the technique used by `keyd`, `xremap`, `input-remapper`, and `libinput` itself.**
- **Prerequisite:** The user must be in the `input` group: `sudo usermod -aG input $USER` (then log out/in). Without this, `/dev/input/` devices are not readable by normal users.

### 5.3 Implementation
- **Removed:** `tauri-plugin-global-shortcut = "2"` from `Cargo.toml` and all usages from `lib.rs` and `capabilities/default.json`.
- **Added:** `evdev = "0.12"` to `Cargo.toml`.
- **New `src-tauri/src/hotkey/mod.rs`** (was empty placeholder):
  - `find_keyboards()` — Enumerates `/dev/input/` and filters devices that report `KEY_SPACE` + `KEY_LEFTCTRL` (reliable keyboard heuristic).
  - `start_listener(callback)` — Spawns one blocking OS thread per keyboard device. Uses `device.fetch_events()` which blocks on `read()` syscall — zero CPU when idle. Tracks Ctrl held-state (shared `Arc<Mutex<bool>>` across all device threads). Fires `callback()` on `KEY_SPACE` press (`value == 1`, not autorepeat `2`) while Ctrl is held.
- **`lib.rs` setup closure:** Replaced shortcut plugin registration with `hotkey::start_listener(move || { std::thread::spawn(...handle_hotkey_press...) })`.
- **Build status:** Compiling at time of log write (fetching evdev crate for first time). Expected: clean build.

## 6. Security Refactor — Migrating from evdev to IPC (Unix Domain Sockets)

### 6.1 Vulnerability Identified
While the `evdev` approach successfully bypassed Wayland's strict window-focus sandbox, it introduced a significant security vulnerability. Adding the user to the `input` group allows *any* process running under the user's account to globally read all keystrokes from `/dev/input/event*`, effectively nullifying Wayland's keyboard sandboxing system-wide.

### 6.2 Solution — Wayland-Native Global Shortcuts via IPC
Instead of having the background daemon read hardware events directly, we shifted the responsibility of global shortcut handling back to the compositor (GNOME/Wayland) where it securely belongs:
- The user registers a Custom System Shortcut natively in their Desktop Settings.
- When triggered, Wayland securely intercepts the keystroke and executes our lightweight CLI command: `whispershell --toggle-recording`.
- We implemented **Unix Domain Sockets** (`std::os::unix::net`) for Inter-Process Communication (IPC). 

### 6.3 Implementation Details
1. **Background Daemon (`src/lib.rs`):** We replaced the `evdev` hardware polling loop with a background thread running a `UnixListener` bound to `/tmp/whispershell.sock`. It passively listens for connections and waits for the `"TOGGLE"` signal to execute the dictation logic.
2. **CLI Client (`src/main.rs`):** If the binary is launched with `--toggle-recording`, it bypasses the heavy Tauri app boot sequence. Instead, it instantly connects to `/tmp/whispershell.sock`, sends `"TOGGLE"`, and immediately exits.
3. **Cleanup:** Removed the `evdev` crate, deleted `src/hotkey/mod.rs`, and updated the UI in `index.html` to display the new CLI trigger command.
4. **Security Restored:** The user can now be safely removed from the `input` group, completely eliminating the keylogger vulnerability while achieving perfect, secure Wayland integration!

## 7. UX Polish & Multi-Monitor Support

### 7.1 Multi-Monitor Overlay Positioning
- **Issue:** The overlay window defaulted to opening on the primary monitor, regardless of which screen the user was actively focused on or had their cursor on.
- **Solution:** Integrated Tauri's `cursor_position()` API to retrieve the exact coordinates of the mouse. Iterated over `available_monitors()` to find the monitor bounding box containing the cursor, ensuring the overlay is always spawned on the user's active screen.
- **Adjustment:** Increased the bottom margin of the overlay from `20px` to `100px` logical units, raising it slightly upwards for better visibility.

### 7.2 System Shortcut Troubleshooting
- **Issue:** The custom system hotkey configured via GNOME settings (using `cd ... && cargo run ...`) silently failed.
- **Root Cause:** Linux desktop environment shortcut runners use `exec()` without invoking a shell. Shell built-ins (`cd`) and operators (`&&`) are invalid in this context. Furthermore, using `cargo run` in a hotkey incurs compilation checks and delays.
- **Solution:** The shortcut must use the absolute path to the compiled binary (`.../target/debug/whispershell --toggle-recording`). This bypasses the shell entirely and provides zero-latency IPC triggering.

### 7.3 Custom Icon Integration
- **Implementation:** Replaced the default Tauri placeholder icons with a custom suite of `whispershell-icons`.
- **Configuration:** Updated `tauri.conf.json` to reflect the exact PNG names and removed legacy references.
- **Frontend Assets:** Implemented the custom SVG (`whispershell-icon.svg`) as the UI logo inside the `overlay.html` HUD, and injected it as a favicon across both frontend HTML files.
- **Asset Sandboxing Fix:** Since Tauri restricts asset loading strictly to the `frontendDist` directory (`ui/`), relative paths like `../icons/` failed to load in the browser. To resolve this, we copied `whispershell-icon.svg` into a new `ui/icons/` directory so it's accessible within the frontend root.

### 7.4 UI Minimalist Theme Refactor
- **Objective:** Streamline the main settings/status window (`index.html`) to align with the "lavender black" aesthetic of the overlay HUD.
- **Changes:** 
  - Removed all emoji icons (`🎙️`, `🔴`, `⚙️`, etc.) to achieve a cleaner, professional look.
  - Replaced the large animated status orb with a minimalist colored indicator dot (`Idle`, `Recording`, `Processing`).
  - Reduced unnecessary paddings, margins, and complex layout structures, opting for a compact, to-the-point card view.
  - Unified color palettes across `overlay.html` and `index.html` using the same lavender dark theme variables (`--bg-base`, `--bg-card`, `--border`, `--accent`).
